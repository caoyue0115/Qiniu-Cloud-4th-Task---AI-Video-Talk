## 1. 产品定位与目标用户

### 1.1 用户与痛点

- **目标用户**：中国 1731 万视障人士（2023 数据），全球约 2.85 亿。
- **核心痛点**：日常生活中「视觉信息获取」是刚需且高频——认不出药品、分不清颜色、读不了说明书、过马路看不见红绿灯、找不到东西。
- **现有方案不足**：Be My Eyes 依赖志愿者（接通率不稳定、占用他人、有"麻烦别人"的心理负担）；Seeing AI 偏 iOS、中文场景弱。
- **政策利好**：《无障碍环境建设法》2023 年实施，鼓励信息无障碍。

### 1.2 产品定位

> **一句话**：让视障者通过纯语音对话，让 AI 帮他们「看」世界——7×24 可用、无需求人、无心理负担。

**差异化**：
1. AI 驱动、随时可用，不依赖志愿者，无"麻烦别人"的负担。
2. **从第一行就为"看不见"设计交互**，而非给视觉 App 加个语音。
3. 端云协同控成本 + 国产化全栈（通义千问），可持续。

### 1.3 核心闭环

```
语音输入 → 意图理解（端侧/小模型/大模型/模式切换）→ 视觉理解 → 语音输出
```

## 2. 用户故事：计划实现 vs 最终实现

### 2.1 用户故事与优先级（MoSCoW）

| 优先级 | 用户故事 | 状态 |
|--------|---------|------|
| Must | US-01 我想问"这是什么"，AI 看图告诉我名称/颜色/用途 | ✅ 已实现 |
| Must | US-02 我想问"这是什么颜色"，AI 准确辨色 | ✅ 已实现 |
| Must | US-03 我想让 AI 读出菜单/说明书/验证码上的文字 | ✅ 已实现（阅读模式 + 问答均可）|
| Must | US-04 全程纯语音操作，不依赖看屏幕 | ✅ 已实现（语音进入/切换/打断/重听/调速）|
| Should | US-05 导航模式：提醒前方行人/车辆/红绿灯/楼梯等危险 | ✅ 已实现（端侧实时检测 + 云端补充）|
| Should | US-06 多轮对话与上下文记忆，可自然追问"这个能吃吗" | ✅ 已实现（共享记忆，看图/聊天互通）|
| Should | US-07 聊天陪伴模式：AI 主动找话题、自然对话 | ✅ 已实现 |
| Could | US-08 实时通话级低延迟对话 | ✅ 已实现（实时模式 Omni-Realtime，超出原计划）|
| Could | US-09 无障碍增强：震动、语速调节、再说一遍、帮助 | ✅ 已实现 |
| Won't(本期) | 地图级目的地导航（接高德等） | ❌ 未做（高德不做视觉识别，与本期互补，列入后续）|

### 2.2 变更说明（体现决策思维）

- **超额完成**：原计划 US-08 仅"Could"，但调研发现"延迟"是视障实时交互的核心体验瓶颈，遂引入 **Qwen-Omni-Realtime 实时模式**（端到端首字约 0.3s），并保留稳定模式作高准确/低成本兜底——形成**双引擎**差异化。
- **架构演进**：从最初的 Gradio 单页 Demo（仅问答）演进为 **FastAPI 通话式应用 + 四大模式 + 双引擎 + 无障碍体系**，并额外封装了一层**艺术化展示网站**供评委直观体验。
- **务实取舍**：地图导航（高德）本期不做——高德只能做 GPS 路线、不能做摄像头视觉避障，二者互补而非替代；本期聚焦"看见"这一核心价值。
- **可靠性优先**：流式 ASR 实测在弱网下 WebSocket 会话易断导致"一直没听清"，故稳定模式回退为更可靠的一次性识别 + 自动重试——宁可慢约 1 秒，也要稳。

## 3. 面向视障人士的专门设计

> 这是本产品区别于"视觉 App 加语音"的核心。视障者**看不见屏幕、靠听与触感知**，每一处交互都为此重新设计。

### 3.1 交互设计：纯语音 + 触觉，零视觉依赖

| 设计 | 为视障者解决的问题 |
|------|-------------------|
| **语音进入/切换模式** | 说「导航模式/阅读模式/聊天模式/退出」即可切换，无需找按钮 |
| **VAD 静音自动触发** | 说完停顿即自动识别，不用按任何键 |
| **语音 + 点击双打断** | AI 朗读时可直接说话打断，或**点屏幕任意处**打断（看不见停止键时的可靠兜底）|
| **震动反馈** | 开始聆听 / 答案就绪 / 危险告警时震动——嘈杂环境听不清也能靠触感知 |
| **语速调节** | 说「说快点/说慢点」，贴合资深盲人偏好快语速的习惯 |
| **再说一遍 / 帮助** | 说「再说一遍」重听漏掉的回答；说「我能做什么」语音列出全部功能 |
| **全程语音引导** | 进入、接通、各环节均有语音提示，屏幕阅读器友好 |
| **无打字框、大按钮、aria-label** | 适配 TalkBack/VoiceOver，按钮去 emoji、加大、带语义标签 |

### 3.2 回答设计：为"听"而答，不为"看"而答

AI 回答规范（写进系统提示词，稳定模式与实时模式统一）：

1. **听觉优先、零视觉指代**：禁用"如图/这边/那个/左边这个"，改用**方位 + 距离 + 可触摸特征**（"你右手边约一臂远""瓶身有凸起纹路"）。
2. **危险/关键信息先说**（倒金字塔）：过马路先说红绿灯，盲人靠听，第一句给结论。
3. **主动补盲人看不到但需要的信息**：颜色、是否过期、数量、空间方位、危险提示。
4. **不确定就说不确定、保守**：盲人无法用眼复核，宁可说"可能是…，拿近点我再看看"，涉及药品/马路更保守。
5. **简短可听**：1–3 句、口语化，因为是语音播报，长了记不住。
6. **可追问引导**：答完主动提示"要不要读一下成分/看保质期"，因为盲人不知道画面里还有什么可问。

### 3.3 安全与场景化

- **导航**：端侧实时检测（行人/车辆/红绿灯）按方位 + 远近**即时**告警 + 震动；云端千问补充楼梯/路况/指示牌。
- **阅读**：行为"框死"为朗读，说「翻页了」自动重读，不会答非所问。
- **药品/验证码**：提示遵医嘱、验证码逐字读。

## 4. 运营成本控制：想到的 vs 实际采用的

> AI 产品的成本贯穿全生命周期。下面区分"想到的技巧"与"本项目实际采用并验证的"。

### 4.1 技巧对照表

| # | 成本控制技巧 | 是否采用 | 本项目做法 |
|---|------------|---------|-----------|
| 1 | **端侧处理简单任务** | ✅ | 颜色识别(HSV)、人/车/红绿灯(TensorFlow.js COCO-SSD)在浏览器本地完成，**零云端成本** |
| 2 | **模型分级路由** | ✅ | 简单问题走 qwen-vl-plus(小模型 ¥0.012)，复杂场景才走 qwen-vl-max(大模型 ¥0.03) |
| 3 | **图像压缩** | ✅ | 上传前压到 512–640px、JPEG 质量 ~0.72，降带宽与视觉 token |
| 4 | **单帧上传** | ✅ | 问答只传"提问那一刻"的 1 帧（而非多帧），手机上传更快、token 更省 |
| 5 | **缓存** | ✅ | 相同问题 + 相近画面命中缓存，零调用 |
| 6 | **本地语音兜底** | ✅ | TTS 优先端侧 speechSynthesis（零成本零延迟），无本地中文语音才回退云端 CosyVoice |
| 7 | **识别与语音并行** | ✅ | 导航/聊天的周期识别不阻塞麦克风，避免无效等待与重复调用 |
| 8 | **实时模式选 flash 档** | ✅ | 实时用 qwen3.5-omni-**flash**-realtime(比 plus 便宜约 3 倍)，并用 100 万 token 免费额度 |
| 9 | **流式 ASR / 蒸馏 / 本地部署** | ⚠️ 部分/未做 | 流式 ASR 弱网不稳已回退；蒸馏与本地小模型部署列入 V2.0 |

### 4.2 成本效果（实测/估算）

**稳定模式**（按次）：
```
颜色识别            端侧   ¥0.000
简单问答(小模型)    云端   ASR0.005 + VL0.012 + TTS(端侧≈0)  ≈ ¥0.017
复杂场景(大模型)    云端   ASR0.005 + VL0.030               ≈ ¥0.035
命中缓存                   ¥0.000
```
> 相比"纯云端 + 大模型 + 多帧"，端云协同 + 分级路由 + 单帧，**单次成本降约 62%**。

**实时模式**（按 token，qwen3.5-omni-flash-realtime，华北2）：
| 项 | 每分钟用量 | 单价 | 成本 |
|----|-----------|------|------|
| 输入音频 | 60s×12.5 ≈ 750 token | 27 元/百万 | ¥0.020 |
| 输入图片 | 512px帧≈192 token，按时长约30帧 | 3.3 元/百万 | ¥0.019 |
| 输出音频 | AI约说40s×12.5 ≈ 500 token | 107 元/百万 | ¥0.054 |
| **合计** | | | **≈ ¥0.09/分钟** |
> 前 90 天有 100 万 token 免费额度，约够 130 分钟实时通话。

**成本可视化**：展示页左侧仪表盘实时显示每次调用走了哪条路径、累计成本、端侧+缓存占比，让"省了多少"一目了然——这本身就是给评委/运营看的成本透明化设计。

## 5. 关键产品决策与取舍

| 决策点 | 取舍 | 理由 |
|--------|------|------|
| 双引擎（稳定 + 实时） | 都保留、可切换 | 稳定=准+省+稳，实时=快+自然；不同场景不同诉求，不是二选一 |
| 颜色识别走端侧还是云端 | 最终走云端 VL | 真实摄像头下端侧 HSV 受白平衡/反光影响不准，云端 VL 更可靠（牺牲一点成本换准确）|
| 流式 ASR vs 一次性 ASR | 选一次性 + 重试 | 流式更快但弱网下易断导致"一直没听清"；可靠性优先 |
| 导航实时性 | 端侧 TF.js + 云端千问混合 | 纯云端 ~3s 延迟做不到实时避障；端侧 COCO-SSD 毫秒级告警，千问补语义 |
| 模型本地化托管 | 把 TF.js/模型权重放自己服务器 | COCO-SSD 默认从被墙的 Google 存储下载，改本地托管保证国内可用 |
| 演示形态 | 包装艺术化展示网站 | 评委只看网页端，用手机样机 + 实时仪表盘直观呈现"省钱"与"看见" |

## 6. 系统架构概览

```
浏览器（手机/电脑）
  · 摄像头/麦克风/扬声器  · VAD 自动触发  · 端侧 COCO-SSD 实时检测
  · 语音/点击打断  · 震动反馈  · 16k 采样
        │ 稳定模式 HTTP/SSE          │ 实时模式 WebSocket
        ▼                            ▼
FastAPI 服务端
  意图路由 → Paraformer ASR / 千问 VL(plus·max) / CosyVoice TTS / Omni-Realtime 中继
  缓存 · 成本追踪 · 对话记忆
        │
        ▼
阿里云百炼 DashScope（通义千问全栈，国产化）
```

详见 [技术实现路线.md](技术实现路线.md)。

---

> **结语**：视障辅助产品的难点不在"能不能用 AI"，而在**能否真正贴合看不见的人的使用方式**，并在有限成本下可持续。本产品在每一处交互、每一句回答、每一次调用上，都为"看不见"与"省成本"做了专门设计。
