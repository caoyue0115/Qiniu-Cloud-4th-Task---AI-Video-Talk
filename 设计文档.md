## 1. 产品定位与核心闭环

**一句话定位**：让视障人士通过语音对话的方式，让 AI 帮他们「看」世界——识别物体、辨别颜色、阅读文字、描述场景、识别验证码。

**核心闭环**：

```
语音输入(ASR) → 意图路由 → 视觉理解(VL) / 端侧CV → 语音输出(TTS)
```

每一环都围绕两个目标设计：**视觉理解的准确性**、**语音交互的自然流畅**，并在路由层贯穿**端云协同的成本控制**。

## 2. 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                      Gradio Web 界面                      │
│              麦克风输入 + 摄像头画面 + 成本面板           │
└───────────────────────────┬─────────────────────────────┘
                            │ process_request(audio, image)
                            ▼
┌─────────────────────────────────────────────────────────┐
│                    编排层 (app.py)                        │
│   1. ASR 转写  2. 缓存查询  3. 路由分类  4. 调用  5. TTS  │
└───┬──────────┬──────────┬──────────┬──────────┬──────────┘
    ▼          ▼          ▼          ▼          ▼
 speech    cache     router    vision/color  cost_tracker
 (ASR/TTS) (L1缓存)  (意图分类) (VL/端侧HSV)  (成本可视化)
                                  │
                            context (多轮上下文)
```

### 模块职责

| 模块 | 文件 | 职责 |
|-----|------|------|
| 编排 | `app.py` | 串联完整闭环，决定每个请求走哪条路径 |
| 语音 | `modules/speech.py` | Paraformer 语音识别 + CosyVoice 语音合成 |
| 视觉 | `modules/vision.py` | 通义千问 VL（max/plus 双档）图像理解 |
| 端侧颜色 | `modules/color_detector.py` | HSV 色彩空间分析，**零 API 成本** |
| 路由 | `modules/router.py` | 关键词意图分类 → edge/text/simple/complex |
| 缓存 | `modules/cache.py` | L1 精确缓存（内存版，生产替换 Redis） |
| 成本 | `modules/cost_tracker.py` | 记录每条路径调用次数与成本，可视化 |
| 上下文 | `modules/context.py` | 多轮对话记忆，限制 token 消耗 |
| 工具 | `utils/image_utils.py` | 图像压缩、base64 编码、帧差计算 |
| 配置 | `utils/config.py` | 从 `.env` 读取 Key 与模型名 |

## 3. 用户故事：计划实现 vs 最终实现

### 3.1 计划实现的用户故事（MoSCoW 优先级）

| 优先级 | 编号 | 用户故事 | 计划版本 | 依赖 |
|-------|------|---------|---------|------|
| Must | US-001 | 日常物品识别与描述 | V1.0 | 视觉模型 |
| Must | US-003 | 颜色识别 | V1.0 | 端侧 HSV |
| Must | US-006 | 多轮对话上下文 | V1.0 | LLM |
| Should | US-005 | 文本阅读辅助 | V1.0 | VL（含OCR能力） |
| Should | US-002 | 障碍物检测 | V1.1 | 端侧 YOLO |
| Could | US-004 | 验证码识别 | V1.2 | 合规确认 |
| Won't | — | 导航辅助 | V2.0 | 地图 SDK |

### 3.2 最终实现状态

| 编号 | 用户故事 | 实现状态 | 实现方式 / 变更原因 |
|------|---------|---------|--------------------|
| US-001 | 物品识别 | ✅ 完整实现 | 通义千问 VL，按问题复杂度走 plus/max |
| US-003 | 颜色识别 | ✅ 完整实现 | 端侧 HSV，含黑白灰判断与暗光提示，零成本 |
| US-006 | 多轮上下文 | ✅ 完整实现 | `context.py`，保留最近 6 轮 + 5 分钟 TTL |
| US-005 | 文本阅读 | ✅ 完整实现 | VL 自带 OCR，路由关键词"念/读/写了什么"触发 |
| US-004 | 验证码识别 | ✅ 基础实现 | VL 可识别，路由含"验证码"关键词；**合规边界见 §5** |
| US-002 | 障碍物检测 | ⏳ 部分实现 | 当前走云端 VL 描述方位；端侧 YOLO 实时检测**未集成**（见下） |
| 导航辅助 | — | ❌ 未实现 | 按计划移至 V2.0 |

### 3.3 变更说明（决策思维）

> **为什么 US-002 障碍物检测只做了一半？**
> 原计划用端侧 YOLOv8-nano 做实时检测以省成本。但实测中低端机推理延迟 >1.5s，
> 无法满足"边走边检测"的实时性。**临时方案**：障碍物类问题走云端 VL，让大模型
> 描述画面中障碍物的方位与距离——牺牲一点成本，换取可用性。
> 端侧 YOLO 的接口已在 `router.py` 预留路径，待 V1.1 用 NPU 优化或更轻的
> MobileNet-SSD 补齐。这体现"**用户体验优先**"的取舍。

> **为什么把 US-005 从 Should 提前到 V1.0 完整实现？**
> 因为通义千问 VL 本身具备 OCR 能力，文本阅读**无需额外集成 OCR 模型**，
> 边际成本几乎为零。能低成本交付的高价值能力，优先做。

## 4. 运营成本控制：想到的技巧 vs 实际采用的

> 成本结构（单次"这是什么"请求估算）：ASR ¥0.005 + 视觉 ¥0.012~0.030 + TTS ¥0.003 ≈ **¥0.02~0.04/次**。
> 若 1 万 DAU、人均 50 次/天，纯云端无优化月成本可达百万级——**成本控制至关重要**。

### 4.1 技巧对照表

| # | 想到的技巧 | 是否采用 | 实际做法 | 预期效果 |
|---|-----------|---------|---------|---------|
| 1 | **端侧处理简单任务** | ✅ 采用 | 颜色识别走端侧 HSV，零云端调用 | 颜色类请求成本→0 |
| 2 | **模型路由分级** | ✅ 采用 | 简单问题→qwen-vl-plus，复杂→qwen-vl-max | 视觉成本降约 50% |
| 3 | **精确缓存** | ✅ 采用 | `cache.py` 按"问题+图像指纹"缓存结果 | 重复请求成本→0 |
| 4 | **图像压缩** | ✅ 采用 | 上传前压到 720p、JPEG q80 | 带宽/token 成本降约 50% |
| 5 | **对话上下文压缩** | ✅ 采用 | 只保留最近 6 轮 + 图像描述复用 | token 消耗降约 40% |
| 6 | **成本可视化监控** | ✅ 采用 | `cost_tracker.py` 实时显示各路径成本与端侧占比 | 让优化效果可量化 |
| 7 | **图像指纹缓存命中** | ✅ 采用 | 降采样后做 md5，相近画面命中同一缓存 | 提升缓存命中率 |
| 8 | **语义缓存（向量）** | ⚠️ 未采用 | 相似场景用向量库召回 | 实现复杂度高，V2.0 再做 |
| 9 | **模型蒸馏** | ⚠️ 未采用 | 蒸馏专用小模型替代大模型 | 实训阶段资源不足，V2.0 |
| 10 | **端侧 YOLO 物体识别** | ⏳ 计划中 | 简单物体端侧识别不走云 | 受限于端侧性能，V1.1 补 |
| 11 | **本地化部署开源模型** | ⚠️ 未采用 | 高并发场景本地部署 Qwen-VL | V2.0 规模化时实施 |
| 12 | **用量限制** | ⚠️ 未采用 | 免费用户每日调用上限 | 上线运营阶段加 |

### 4.2 实际采用策略的成本效果

```
未优化（纯云端、无路由）：单次 ≈ ¥0.12
已优化（端云协同+路由+缓存+压缩+上下文压缩）：单次 ≈ ¥0.045
降幅 ≈ 62.5%

V2.0 目标（+模型蒸馏+本地部署）：单次 ≈ ¥0.02
```

**核心洞察**：成本控制不是单点优化，而是"端侧拦一层、路由分一层、缓存挡一层、压缩省一层"的**漏斗式叠加**。`cost_tracker` 面板让每一层的贡献都可量化——这正是答辩时"用数据支撑决策"的素材。

## 5. 关键决策与取舍

1. **为什么用国产模型（通义千问）而非 GPT-4o？**
   成本更可控（VL-plus 约为 GPT-4o 的 1/5）、中文场景优化好、合规无障碍、有免费额度便于实训验证。架构上 `config.py` 把模型名抽出，可随时切换或加多厂商兜底。

2. **为什么 ASR+VL+TTS 分级管线，而非 Qwen-Omni 全模态？**
   分级管线每一环成本独立可控、可分别缓存、便于在路由层做端云协同；全模态虽自然但成本高、难以分级降本。对成本敏感的本赛题，分级是更优解。

3. **验证码识别的合规边界**：技术可行，但部分网站验证码使用条款限制自动识别。当前仅作基础能力，正式上线前需法务评估——文档中明确标注，体现"合规优先"。

## 6. 运行方式

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 配置密钥：复制 .env.example 为 .env，填入百炼 API Key
#    DASHSCOPE_API_KEY=sk-xxxx
#    申请地址：https://bailian.console.aliyun.com/

# 3. 启动
python app.py
# 浏览器自动打开 http://127.0.0.1:7860
```

**试用问法**：
- 「这是什么颜色」→ 端侧处理，零成本
- 「这是什么」「这是什么药」→ 小模型
- 「帮我看看周围」「前面有障碍物吗」→ 大模型
- 「这上面写了什么」「帮我读验证码」→ 文字识别
